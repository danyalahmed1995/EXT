# Contributing to EXT

Thanks for taking the time to improve EXT.

EXT is a local-first Markdown and text workspace. Contributions should keep that scope intact: fast filesystem access, clean editing, reliable preview, careful file handling, and a polished desktop experience.

## Before opening a pull request

1. Read `DESIGN.md`.
2. Keep the change focused.
3. Test the change manually.
4. Run the relevant checks.
5. Explain what changed and why.

A good pull request does one thing clearly.

## Development checks

Frontend:

```bash
npm run test
npm run build
```

Backend:

```bash
cd src-tauri
cargo fmt --check
cargo clippy -- -D warnings
cargo test
```

Run any additional scripts defined in `package.json` if your change touches those areas.

## Pull request guidelines

Your pull request should include:

- a clear summary
- screenshots or short recordings for UI changes
- reproduction steps for bug fixes
- notes about any risk or follow-up work
- confirmation that you tested the change

Keep diffs small when possible. Large rewrites are harder to review and easier to get wrong.

## Project scope

EXT welcomes improvements to:

- local workspace scanning
- Markdown and text editing
- preview rendering
- file management
- search and smart views
- themes and accessibility
- settings
- Tauri/Rust filesystem safety
- performance
- tests and documentation

EXT is not currently looking for:

- accounts
- cloud sync
- hosted documents
- collaboration
- plugin marketplaces
- AI agents
- analytics or telemetry
- publishing platforms
- proprietary note databases

Feature requests outside the current scope may be closed.

## AI-assisted work

AI tools are allowed. Unreviewed AI output is not.

If you use Copilot, ChatGPT, Claude, Gemini, Codex, or another assistant, you are responsible for the final result. You must understand the code, test it, and be able to explain it during review.

Do not submit:

- pull requests generated by an autonomous agent without manual review
- code you cannot explain
- broad rewrites produced without understanding the existing architecture
- fake bug reports
- generic feature requests copied from an assistant
- mass-generated comments on issues or pull requests

Low-effort AI-generated issues or pull requests may be closed without review. Repeated spam may result in being blocked from the repository.

## Bug reports

A useful bug report includes:

- operating system and version
- EXT version or commit
- steps to reproduce
- expected behavior
- actual behavior
- screenshots if relevant
- logs or terminal output if available

Please do not paste private document contents into an issue.

## Feature requests

A useful feature request explains:

- the workflow problem
- why it fits EXT's local-first scope
- what a small version of the feature could look like
- what should remain out of scope

The best features make the app more useful without making it heavier.

## Security issues

If you find a path traversal issue, unsafe file operation, preview sanitization issue, or another security problem, do not post a public proof-of-concept that exposes users.

Open a minimal issue describing the risk, or contact the maintainer privately if a private channel is provided.

## Review standards

Maintainers may ask for:

- smaller commits
- clearer naming
- tests
- simpler logic
- safer error handling
- screenshots for UI changes
- a different approach that better matches the architecture

That is normal review, not rejection of the contributor.

## Code style

Prefer:

- clear names
- small functions
- typed data
- explicit error handling
- cleanup for listeners, watchers, and timers
- local-first behavior
- boring reliable code

Avoid:

- hidden side effects
- unnecessary dependencies
- clever abstractions
- broad try/catch blocks that hide bugs
- unsafe filesystem shortcuts
- changes that make typing or file browsing slower

## Documentation

Update documentation when your change affects setup, architecture, contribution rules, user behavior, or safety expectations.

## Releases

Release artifact versions are synced directly from Git tags. 
To create a new release:

1. Create a tag in the format `vX.Y.Z` (e.g., `v0.1.2`).
2. Push the tag to origin.

```bash
git tag v0.1.2
git push origin v0.1.2
```

The GitHub Actions workflow will automatically read the tag, sync the version across `package.json`, `tauri.conf.json`, and `Cargo.toml`, and generate release artifacts with the corresponding version (e.g., `EXT_0.1.2_aarch64.dmg` and `EXT_0.1.2_x86_64.dmg` for macOS). Local builds will continue to use the version in `package.json` unless the `RELEASE_VERSION` environment variable is set.
